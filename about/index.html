Spine Event Engine is a framework for building modern cloud applications and aims to free developers from creating boilerplate code as much as possible. Our framework is built on top of Protocol Buffers (a.k.a. protobuf) and gRPC to bring easiness and efficiency into development of applications with microservice-oriented architecture.

The primary language of the backend is Java. Client applications and server-side modules can be written on JavaScript, Java, JavaNano (Android), Swift, Objective-C, C, C++, C#, Go, Python, Ruby, and PHP.

The main usage scenarios:

  • Connecting browser clients and mobile devices to backend services.

  • Maintaining complex and evolving business logic.

  • Integrating multi-platform and multi-language backend systems.

Core Features that make our framework awesome:

  • Code Generation for multiple languages and platforms. Spine makes a domain model language Ubiquitous not only in people interactions, but in computer communications, too. The model is defined and maintained in Protobuf. Most of the domain model code is automatically generated for all the programming languages of your project.

  • Strongly Typed Domain Model. Unlike some CQRS/ES frameworks which treat commands and events as Json objects, Spine promotes strong typing of all parts of a business model. The model can be extended without breaking binary compatibility with client applications.

  • Promoted Immutability. Messages like commands or events, entity state objects, query results, and many other business logic objects being immutable greatly simplify the code since they are automatically thread-safe, and as such are freely shared or referenced.

  • Isolation of development responsibilities. The framework is based on CQRS pattern, which allows to develop write- and read- sides of the backend independently.

  • Easier scaling. Thanks to CQRS pattern, read- and write- sides of the backend can be scaled independently, which reduces maintenance and license costs.

  • Tracability and debuggability. With Event Sourcing built into the framework, it is easier to analyze the history of business data evolution, and find causes if something goes wrong.

  • Choice of Storage and Deployment Platforms. The framework promotes writing storage- and platform-agnostic code. You can start with JDBC and later switch to Google Cloud Platform Datastore by changing few lines of code.

  • Easier and Faster Testing. Since the code is storage-agnostic, unit tests are written and executed on an in-memory storage, which is much faster. Integration tests run for both in-memory and real storage which allows to spot possible timing issues.

  • Open Source. Permissive Apache License allows using the framework in closed-source projects. You are welcome to contribute to the framework development.
